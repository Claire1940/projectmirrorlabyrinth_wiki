<Section icon={Map} title="project mirror labyrinth trello のボード概要">

Project Mirror Labyrinth Trello は、放置用の置き場ではなく、Wiki の指令室として機能させるのが最適です。ボードには、何を調査しているか、何を執筆しているか、何をまだファクトチェックする必要があるか、そして何が公開準備完了かを表示するべきです。この構成により、長いコメントのやり取りを追わなくても、チームは前に進めます。

| ボード領域 | 最適な用途 | 例のカード |

<Section icon={Target} title="セットアップ手順とラベル体系">

強い Trello 構成は、少しずつ層を重ねて作ります。まずボードの形を決め、次にラベルを追加し、それから各カードに必要な内容を定義します。この順序なら、見た目は派手でも実作業を追跡できないボードになってしまう、よくある失敗を避けられます。

<Steps>
  <Step num="1" title="基本リストを作成する">
    まずは Research、Draft、Review、Publish の4つのリストを使います。追加リストは、チームが必要な理由を具体的に説明できる場合にのみ増やしてください。
  </Step>
  <Step num="2" title="ラベルの配色を定義する">
    ラベルは細かな状態ではなく、コンテンツ種別に使います。たとえば、クエストページ、ボスページ、アイテムページ、マップページを分けます。
  </Step>
  <Step num="3" title="1つのカードテンプレートを決める">
    すべてのカードには、短い要約、担当者、ソースリンク、明確な完了条件を入れるようにします。
  </Step>
  <Step num="4" title="レビューのリズムを固定する">
    週次の監査時間を決めて、ボードを最新の状態に保ち、動いていないカードがアクティブなリストを圧迫しないようにします。
  </Step>
</Steps>

| ラベル | 意味 | 最適な例 |
|---|---|---|
| **Quest** | ストーリーまたはミッション系のコンテンツ | メインクエストページの下書き |
| **Boss** | 敵または遭遇戦のコンテンツ | レイドボス項目 |
| **Item** | 装備、ドロップ、素材 | レアドロップ記録 |
| **Map** | ロケーション、エリア、ルートのコンテンツ | ゾーン構成メモ |
| **Pending** | 検証待ち | 未確認のステータス確認 |

<Callout type="warning" title="ラベルに関する注意">
ラベルをリストの代わりに使わないでください。ラベルはカードが何であるかを示し、リストはカードがワークフローのどこにあるかを示すべきです。
</Callout>

</Section>

<Section icon={Shield} title="カードテンプレート、担当者、確認項目">

ボードの構成が整ったら、各カードにも同じ最低限の情報が必要です。その一貫性こそが、Trello を単なるタスクボードから信頼できる制作システムへ変えます。良いカードテンプレートは、すべての編集者が同じ項目を同じ順番で見られるため、引き継ぎも速くなります。

<Callout type="info" title="カードテンプレートのルール">
各カードは再利用できるページブリーフとして扱いましょう。チームメイトがカードを開いて10秒でタスクを理解できるなら、そのテンプレートは機能しています。
</Callout>

| 項目 | 記入内容 | 重要な理由 |
|---|---|---|
| タイトル | ページ名またはタスク名 | 検索と並べ替えがしやすくなる |
| 要約 | 1行での目的 | カードの焦点を保てる |
| 担当者 | 責任者を1人 | 引き継ぎの曖昧さを防ぐ |
| ソース | リンク、メモ、スクリーンショット | 事実確認を速くする |
| 完了条件 | 明確な完了ルール | 中途半端な作業を防ぐ |

<Section icon={Trophy} title="ワークフロー、公開、保守">

良いワークフローは、派手なボードよりも重要です。次の段階が本当に準備できたときだけカードを移動し、Publish リストはすでにレビューを通過した作業だけに使いましょう。この規律が、ボードの信頼性を保ち、見せかけの進捗からチームを守ります。

| ステージ | 進入条件 | خروج条件 |
|---|---|---|
| Draft | 調査が執筆開始に十分そろっている | 本文の骨格が完成している |
| Review | 下書きが読みやすく構成されている | 事実、リンク、用語が確認済み |
| Ready | レビュアーがカードを承認する | 予定済み、または公開済み |
| Archive | タスクが完了、または廃止された | 参照用にカードを保存する |

| レビュー頻度 | 最適な用途 | 結果 |
|---|---|---|
| 毎日確認 | 進行中のスプリント | 問題の早期発見 |
| 週次監査 | 継続的な Wiki 作業 | バックログの整理 |
| 月次整理 | 大規模ボードの保守 | 古いカードの削減 |

2026-07-06 時点で、現在の Trello の挙動を確認するのに最も有用な公式資料は [Atlassian's Trello Docs](https://support.atlassian.com/trello/docs/) と [Using labels in Trello](https://support.atlassian.com/trello/docs/using-labels-in-trello/) です。ボード運用を標準化する前に、これらで機能の挙動を確認しておくとよいでしょう。

</Callout>

<Section icon={Sparkles} title="よくある問題と FAQ">

小さなボードの問題は、たいてい曖昧なカード名、ラベルの使いすぎ、またはステージが多すぎることから起こります。早めに直せば、ボードは長期的にずっと保守しやすくなります。

| 問題 | 主な原因 | 最適な対処 |
|---|---|---|
| カードが見失われる | リストの範囲が広すぎる | ワークフロー段階ごとに分ける |
| ラベルが散らかって見える | 状態とカテゴリが混在している | ラベルはコンテンツ種別に限定する |
| レビューが止まる | 担当者や期限がない | 担当者1人と期限1つを設定する |
| 重複タスクが出る | 命名が一貫していない | 共通のカードタイトル形式を使う |

<Callout type="tip" title="保守のヒント">
ボードが騒がしく感じるなら、カードを増やす前に複雑さを減らしましょう。よりシンプルなボードのほうが、たいていは速く回せます。
</Callout>
